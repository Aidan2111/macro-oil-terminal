## 4. Backtest

```
POST /api/backtest {z_threshold:2.0, lookback_days:90, start_date:"2024-01-01"}

var_95:        -2701.98
es_95:         -5001.94
es_975:        -5701.94
max_drawdown:  -5701.94
n_trades:      31
sharpe:        4.38
sortino:       31.00
calmar:        15.53
```

Three of the four risk metrics are mathematically distinct (was
all-equal before #71). On this particular fixture `es_975` happens
to coincide with `max_drawdown` — that's a degenerate-distribution
case where the worst single trade *is* the peak-to-trough drawdown.
The unit test in `test_backtest_risk_metrics.py::test_var95_es95_es975_max_dd_are_distinct_on_real_blotter`
covers a non-degenerate fixture where all four are 4-distinct.

## Follow-ups filed

- **#91** (NEW): the Iran STEO `latest_kbpd` returned `3.27` for
  `2026-03`, but Iran's actual crude production is ~3,300 thousand
  bbl/day, not 3.27. The STEO API is publishing the value in millions
  of bbl/day rather than thousands; our wrapper labelled the field
  `kbpd` based on the EIA series-name suffix. Fix: read the STEO
  response unit metadata or rename the field to `mmbpd` and update
  the LLM-context label accordingly. Filed as a non-blocking
  follow-up since the value itself is correct, only the unit label
  is misleading.
