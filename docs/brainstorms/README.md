# Brainstorms

Rough problem explorations. Each file is named `<feature>.md`.

A brainstorm doc answers, before any code is written:

- What is the user actually trying to do?
- What alternatives were considered?
- Why is the chosen path better?
- What assumptions are we making?
- What's unknown, and what would prove us wrong?

It's Socratic — one question per paragraph — and chunked. A teammate should
be able to read it in 5 minutes and disagree specifically.

A brainstorm doc is **not** a spec. It's the raw thinking that a spec is
distilled from. The spec lives in `docs/designs/<feature>.md`.

See `docs/workflow.md` for the full process.
